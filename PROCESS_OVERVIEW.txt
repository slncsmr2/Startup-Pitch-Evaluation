Startup Pitch Evaluation - Process Overview

This document explains what has been implemented in each major process of the project, and how each process works end to end.

1) System Entry Processes

FastAPI process
- Implemented in: backend/app/main.py
- What it does:
  - Starts the API service.
  - Exposes health, single evaluation, batch evaluation, and video listing/streaming endpoints.
  - Serves the frontend page from backend/app/static.
- Why it matters:
  - Gives one HTTP interface for UI and external clients.

Shared inference service process
- Implemented in: backend/app/services/inference.py
- What it does:
  - Builds a stable request_id from payload hash.
  - Runs the same evaluation flow for API and CLI.
  - Provides infer_video helper to turn a video path into a complete PitchInput.
- Why it matters:
  - Keeps API and CLI behavior consistent.

2) Pre-processing and Temporal Segmentation

Temporal synchronization process
- Implemented in: backend/app/services/preprocessing.py
- What it does:
  - Resolves transcript source (payload transcript first, then video transcript_text).
  - Resolves slide context from slides or slide_text.
  - Resolves media file location using configured lookup directory.
  - Splits the full pitch into timeline chunks (default 5 seconds from SPE_CHUNK_WINDOW_SECONDS).
  - Produces PitchChunk objects with:
    - chunk timeline (start/end)
    - chunk text
    - audio/video references
    - alignment metadata
- Why it matters:
  - Creates the canonical chunk-level timeline used by all downstream models.

Local transcription process
- Implemented in: backend/app/services/transcriber.py and preprocessing integration.
- What it does:
  - Supports three backend modes via SPE_TRANSCRIBER_BACKEND:
    - faster-whisper
    - openai
    - auto (prefer faster-whisper, fallback to OpenAI)
  - Applies deterministic fallback states:
    - low quality
    - silence
    - missing audio file
    - empty transcript
  - Returns structured status fields (backend, status, reason, confidence).
- Why it matters:
  - Maintains robust behavior when local AV quality is poor or dependencies are unavailable.

3) Audio and Video Extraction Processes

Video extraction process
- Implemented in: backend/app/services/video_processor.py
- What it does:
  - Extracts sampled frames per chunk when enabled.
  - Uses OpenCV and optional MediaPipe to compute visual metadata:
    - face_ratio
    - motion_score
    - eye_contact_score
    - pose_ratio
    - gesture_energy
  - Persists extracted frames in outputs/frames.
  - Falls back with deterministic extraction_status if video/dependencies are missing.
- Why it matters:
  - Provides visual evidence used by visual scoring.

Audio extraction process
- Implemented in: backend/app/services/audio_processor.py
- What it does:
  - Extracts chunk WAV audio using ffmpeg (system ffmpeg or imageio-ffmpeg fallback).
  - Computes waveform-based quality features:
    - silence_ratio
    - clipping_ratio
    - speech_density
    - pitch_variation
    - energy_variation
    - audio_quality_score
  - Persists extracted audio in outputs/audio_chunks.
  - Falls back with deterministic extraction_status when unavailable.
- Why it matters:
  - Produces chunk-level audio reliability and prosody inputs.

4) Feature Extraction Processes

Text feature process
- Implemented in: backend/models/text_encoder.py and backend/app/services/extractors.py
- What it does:
  - Detects language (en, ta, ta-en) using script ratios, lexical evidence, and langdetect fallback.
  - Produces six text metric signals:
    - Problem Clarity
    - Market Opportunity
    - Solution Uniqueness
    - Traction Evidence
    - Business Model Strength
    - Team Readiness
  - Supports two modes:
    - Heuristic mode: deterministic hashed embeddings + rule-based scores.
    - Neural mode: sentence-transformers embedding + lightweight MLP head.
- Why it matters:
  - Anchors core business/story quality scoring.

Visual feature process
- Implemented in: backend/models/visual_encoder.py and backend/app/services/extractors.py
- What it does:
  - Supports heuristic and neural paths.
  - Heuristic path uses visual metadata and slide/stage context.
  - Neural path uses MobileNetV3-small feature backbone + projection + scoring MLP.
  - Produces:
    - Delivery Clarity
    - Presenter Confidence
- Why it matters:
  - Converts presenter presence and visual execution into investor-readable signals.

Audio feature process
- Implemented in: backend/models/audio_encoder.py and backend/app/services/extractors.py
- What it does:
  - Supports heuristic and neural paths.
  - Heuristic path uses text pace + extracted waveform quality proxies.
  - Neural path uses MFCC + CNN + scoring MLP.
  - Produces:
    - Voice Pace
    - Voice Prosody
- Why it matters:
  - Captures communication dynamics that affect pitch confidence.

5) Fusion and Scoring Processes

Modality fusion process
- Implemented in: backend/models/fusion_head.py and backend/app/services/fusion.py
- What it does:
  - Combines text, visual, and audio embeddings.
  - Outputs fused vector and attention weights for each modality.
  - Heuristic mode uses embedding-energy weighting.
  - Neural mode uses learned cross-modal attention.
- Why it matters:
  - Produces one multimodal representation per chunk while preserving explainable attention.

Chunk scoring process
- Implemented in: backend/models/scoring_head.py and backend/app/services/scoring.py
- What it does:
  - Maps fused and per-modality outputs into 10 quantitative rubric metrics.
  - Produces:
    - text_metrics list
    - av_metrics list
    - aggregate score
    - confidence score
  - Heuristic mode computes weighted aggregate from text/AV/fusion signals.
  - Neural mode uses learned scoring network and optional checkpoint load.
- Why it matters:
  - Converts model outputs into investor-facing scorecard values.

6) Risk, Feedback, and Dashboard Processes

Risk flag process
- Implemented in: backend/app/services/risk.py
- What it does:
  - Adds rule-based flags such as:
    - low-quality-signal
    - competitive-landscape-missing
    - overclaim-risk
    - traction-evidence-weak
- Why it matters:
  - Highlights practical diligence concerns beyond numeric score.

Feedback and reporting process
- Implemented in: backend/app/services/reporting.py
- What it does:
  - Generates strengths, weaknesses, and suggestions from score patterns and risk flags.
  - Builds dashboard series for:
    - quantitative_scores
    - modality_weights
    - risk_distribution
- Why it matters:
  - Turns scoring outputs into actionable guidance.

7) Pipeline Orchestration Process

End-to-end evaluator process
- Implemented in: backend/app/pipeline.py
- What it does:
  - Runs preprocessing to build chunks.
  - Uses thread pool to extract text, visual, and audio features per chunk in parallel.
  - Applies fusion, scoring, risk detection, and report construction for each chunk.
  - Aggregates chunk outputs into final summary:
    - overall_score
    - confidence_score
    - investment_band
    - dominant language_detected
    - processing_option and processing_notes
- Why it matters:
  - This is the central runtime path used for all inference.

8) Neural Network Training and Evaluation Processes

Dataset loading process
- Implemented in: backend/training/dataset_loader.py
- What it does:
  - Loads Option A JSONL splits (train/val/test).
  - Supports legacy format fallback.
  - Derives missing investment_band from overall score.
  - Uses synthetic sample fallback when split files are absent.
- Why it matters:
  - Allows training to run in both real-data and fallback environments.

Training process
- Implemented in: backend/training/trainer.py and backend/scripts/train.py
- What it does:
  - Loads config from models/config/training_cpu.yaml or training_gpu.yaml.
  - Builds multimodal training model with:
    - text/visual/audio encoders to common dimension
    - attention fusion
    - metric head (10 scores)
    - overall head
    - investment band head (3 classes)
  - Optimizes combined objective:
    - metric regression loss
    - classification loss
    - ranking proxy loss
  - Uses AdamW, cosine warm restarts, optional mixed precision, and early stopping.
  - Saves best checkpoint to models/checkpoints.
- Why it matters:
  - Produces the learned neural scoring artifact.

Evaluation process
- Implemented in: backend/training/trainer.py and backend/scripts/evaluate.py
- What it does:
  - Loads checkpoint and evaluates on test split.
  - Reports:
    - MAE
    - RMSE
    - Spearman rank correlation
    - band accuracy
    - loss
- Why it matters:
  - Quantifies model performance before deployment/use.

9) Runtime Benchmark Process

Latency benchmark process
- Implemented in: backend/scripts/benchmark_runtime.py
- What it does:
  - Runs repeated inference calls for cpu and gpu profiles.
  - Reports avg, p95, min, max and per-run timings.
  - Writes outputs/benchmark_runtime.json.
- Why it matters:
  - Tracks performance baseline and regression over time.

10) Dataset Preparation Process

Whisper preparation process
- Implemented in: dataset_tools/prepare_videos_with_whisper.py
- What it does:
  - Batch-discovers videos from a folder.
  - Runs local faster-whisper transcription (with optional CUDA).
  - Falls back to CPU int8 automatically when CUDA runtime is unavailable (unless strict mode is on).
  - Produces two JSONL outputs:
    - pitch_inputs.jsonl for inference
    - labeling_input.jsonl for rubric labeling pipeline
- Why it matters:
  - Speeds up preparing real pitch datasets for both inference and training-label generation.

11) Configuration Process

Environment and runtime configuration
- Implemented in: backend/app/core/config.py
- What it does:
  - Loads SPE_ prefixed settings from root .env and backend/.env.
  - Controls:
    - heuristic vs neural mode
    - chunk window size
    - transcriber backend and quality threshold
    - media lookup directory
    - neural checkpoint and encoder/backbone choices
- Why it matters:
  - Central point to switch system behavior without code edits.

12) Testing Process

Automated test process
- Implemented in: backend/tests
- What it covers:
  - API behavior
  - pipeline output shape
  - fallback scoring
  - language detection
  - media processor fallback behavior
  - transcriber behavior
- Why it matters:
  - Protects the core scoring flow from regressions.

13) Practical End-to-End Flow

1. Client sends payload (or CLI builds payload from video).
2. Preprocessing resolves transcript/slides/media and builds timeline chunks.
3. Audio/video processors create chunk-level metadata and assets.
4. Extractors generate text, visual, and audio features.
5. Fusion head computes modality attention and fused representation.
6. Scoring head outputs 10 metrics + aggregate/confidence.
7. Risk rules add diligence flags.
8. Reporting builds investor dashboard and feedback.
9. API/CLI returns structured evaluation response.

14) Current State Summary

- Heuristic path is the reliable default production path.
- Neural components are integrated and configurable, with checkpoint-based training/evaluation workflows in place.
- Fallback behavior for media and transcription is implemented for robustness.
- Dataset tooling and benchmarking workflows are implemented for iteration speed and observability.
